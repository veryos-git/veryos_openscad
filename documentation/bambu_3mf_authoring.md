# Authoring multi-plate Bambu Studio .3mf files

Notes from reverse-engineering Bambu's .3mf format while writing
`projects/magnet_wall_mount/build_3mf.py`. Save yourself the iterations.

## Anatomy

A `.3mf` is a ZIP. For Bambu Studio to open it as a *project* (not just
geometry), it needs these entries:

```
[Content_Types].xml                   MIME types; fixed boilerplate
_rels/.rels                           rel from package to /3D/3dmodel.model
3D/3dmodel.model                      top-level: outer objects + <build><item>
3D/_rels/3dmodel.model.rels           rels to each per-object .model file
3D/Objects/object_N.model             one per STL; holds the mesh
Metadata/model_settings.config        plater_name, per-plate routing, parts
Metadata/project_settings.config      printer/filament/process JSON (copy from template)
Metadata/slice_info.config            client version header
Metadata/cut_information.xml          one <object> per outer id
Metadata/filament_sequence.json       {"plate_1": {"sequence": []}, ...}
```

## Gotchas (each cost an iteration)

### 1. Inner object IDs must be globally unique

Every `object_N.model` has an `<object id="X">` inside. If all of them use
`id="1"`, Bambu deduplicates meshes and renders the same one on every plate
(plate names change, geometry doesn't). Use unique odd IDs (1, 3, 5, ...)
and reference them from the main `<component objectid="X">`.

### 2. Plates are routed by world (X, Y) position, not by metadata

The `<plate><model_instance><metadata key="object_id">` block in
`model_settings.config` is *metadata only*. The `<build><item>` transform's
translation determines which plate an object actually lands on. Place each
object at its target plate's bed centre.

### 3. Bambu's multi-plate world layout (A1 mini)

Derived empirically by saving a manual 30-plate file and reading
`<item transform>` positions:

- 6 columns wide, rows wrap below
- Stride: **215 mm** in both axes
- Columns extend in **+X**, rows extend in **-Y** (towards printer front)
- Plate 1 bed centre at world (90, 90)
- Plate N bed centre = `(90 + col*215, 90 - row*215)` where
  `col = (N-1) % 6`, `row = (N-1) // 6`

Other printers will have different strides and grid widths. Derive
empirically by saving a manual file and inspecting the build item transforms.

### 4. Build transform mirrors Y and Z

```
transform="1 0 0 0 -1 0 0 0 -1 X Y bH/2"
```

The 3MF transform is column-major 4x3. Bambu's convention flips Y and Z.
The mesh must be pre-centred at the origin; `bH` is its Z bbox height; the
Z translation is `bH/2` so the bottom sits on the bed.

### 5. Settings inheritance

Bambu Studio writes all printer/filament/process settings into
`Metadata/project_settings.config` (a single JSON file). Copying that one
file from a Bambu-saved template into the output preserves them.

Older or library-saved profiles may also include
`Metadata/process_settings_*.config`, `filament_settings_*.config`,
`printer_settings_*.config` — copy those too if present.

### 6. Thumbnails are optional

`Metadata/plate_N.png`, `top_N.png`, `pick_N.png` etc. in the example file
are regenerated by Bambu on slice. Safe to omit.

## Recipe: deriving plate positions for a new printer

1. Open Bambu Studio with the target printer profile.
2. Add 1 object, right-click -> Add Plate, repeat until you have ~5-10 plates.
3. Drag one copy of the object onto each plate (Bambu won't auto-centre
   exactly, but close enough).
4. Save as `.3mf`.
5. Unzip, open `3D/3dmodel.model`, look at the `<item ... transform="...">`
   translations. Cluster them by row/col to find stride and direction.

## Recipe: 15 plates of the same object at varying scale

Two approaches:

**A. Bake scale into each mesh.** Load the STL once, multiply
`mesh.vertices` by scale factor `s`, write 15 different `object_N.model`
files. Recompute `bH` after scaling so the Z offset stays correct. Pro:
trivial extension of current script. Con: 15x geometry in the archive.

**B. One shared mesh, scaled component transforms.** Write a single
`object_1.model`. Use 15 outer objects whose `<component>` references it
with a scaling transform `s 0 0 0 s 0 0 0 s 0 0 0`. Pro: small archive.
Con: the build item Z translation and `source_offset_z` in
`model_settings.config` must be scaled too (`s * bH/2`); verify Bambu
honors component-level scale (test with a 2-plate file first).

Start with A unless archive size matters.

## Reference implementation

`projects/magnet_wall_mount/build_3mf.py` -- packs N STLs into an N-plate
.3mf, one STL per plate. CLI:

```
python3 build_3mf.py -i ./stl_folder -o ./out.3mf -t ./template.3mf
```

`-t` is optional; it copies printer/filament/process settings from a
Bambu-saved reference .3mf.
